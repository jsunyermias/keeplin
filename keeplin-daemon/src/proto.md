# `src/proto.rs` — generated Protocol Buffers code

## Purpose

This file includes the Rust source code that was automatically generated from
`proto/keeplin.proto` by `tonic-build` at compile time. It makes all generated message
types and service traits available under the path `crate::proto::keeplin::*` for the
rest of the daemon crate.

## How it works

1. `build.rs` runs `tonic-build` on `proto/keeplin.proto`.
2. `tonic-build` invokes `protoc` and then generates Rust code, writing it to a file
   named `keeplin.rs` inside Cargo's `OUT_DIR` (a temporary directory whose path varies
   per build).
3. The `tonic::include_proto!("keeplin")` macro expands to `include!(concat!(env!("OUT_DIR"), "/keeplin.rs"))`,
   which splices the generated file into this module at compile time.

## Module structure

```rust
pub mod keeplin {
    // All generated types and traits live here.
    // Examples:
    //   keeplin::Note
    //   keeplin::keeplin_service_server::KeeplinService  (the trait to implement)
    //   keeplin::keeplin_service_server::KeeplinServiceServer  (the tonic wrapper)
    //   keeplin::SyncProgress
    //   keeplin::sync_progress::Stage
}
```

## Lint suppressions

The module is annotated with `#[allow(dead_code, clippy::all)]`:
- `dead_code` — generated code often contains fields or methods not used directly by
  the daemon (e.g. the generated client stubs, which exist for testing purposes).
- `clippy::all` — the generated code does not follow the same style conventions as
  hand-written code; suppressing clippy here avoids noise in CI.

## Design notes

- Do not edit this file manually. Any change would be overwritten on the next
  `cargo build` run.
- To change the gRPC API, edit `proto/keeplin.proto` and rebuild. The Rust types and
  traits in this module update automatically.
- The `build_client(true)` flag in `build.rs` ensures that client-side stubs are also
  generated; they are useful for writing integration tests that call the daemon over
  a real gRPC channel.
